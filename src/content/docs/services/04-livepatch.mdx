## Livepatchとは

**Kernel Livepatch** は、**Linuxカーネルの重大なセキュリティ脆弱性を、システムを再起動せずにパッチ適用する** 仕組みです。

```bash
livepatch        yes       enabled      Canonical Livepatch service
```

家庭用PCでは **enabled（有効）** にしておくのが推奨です。

---

## なぜ再起動が問題なのか

### 通常のカーネル更新

```
通常のカーネルセキュリティ更新:

1. apt upgrade でカーネルパッケージが更新される
2. 新しいカーネルはディスクにインストールされる
3. しかし、現在メモリ上で動いているのは古いカーネル
4. 再起動するまで、新しいカーネルは適用されない！

   ┌────────────┐
   │  メモリ上    │ ← まだ古い脆弱なカーネルが動作中
   │ (旧カーネル) │
   └────────────┘
   ┌────────────┐
   │  ディスク上  │ ← 新しいカーネルは待機中
   │ (新カーネル) │
   └────────────┘

   → 再起動するまでの間、脆弱性は放置状態 😱
```

### Livepatchの場合

```
Livepatchによるカーネル更新:

1. 脆弱性が報告される
2. Canonicalがliveパッチを開発
3. パッチが動作中のカーネルメモリに直接適用される
4. 再起動不要！

   ┌────────────────────────┐
   │  メモリ上のカーネル       │
   │  ┌──────────────────┐  │
   │  │ 脆弱な関数         │  │
   │  │ → パッチで上書き ✅ │  │
   │  └──────────────────┘  │
   └────────────────────────┘

   → 再起動なしで脆弱性が修正される 🎉
```

## Livepatchの仕組み

### ホットパッチの技術的な流れ

```
1. Canonicalのセキュリティチームが脆弱性を分析
   └─ 対象: Critical / High の CVSSスコアを持つカーネル脆弱性

2. パッチの開発
   └─ 脆弱な関数の修正版を作成

3. テスト
   └─ 実機でテスト（エミュレーションではない）
   └─ 過去の累積パッチとの互換性テスト

4. 配信
   └─ Livepatch Client がパッチをダウンロード
   └─ カーネルのメモリ上で脆弱な関数を新しい関数に差し替え

5. 完了
   └─ 再起動なしで脆弱性が修正される
```

### 累積パッチ方式

:::note[安全性の設計]
Livepatchは「増分パッチを積み重ねる」方式ではなく、**累積パッチ**方式を採用しています。

- ❌ パッチA → パッチB → パッチC（積み重ね = 不安定になりやすい）
- ✅ 常に最新の1つの累積パッチを適用（安定性が高い）

各パッチは過去のすべてのパッチを含んだ状態でテストされています。
:::

## Livepatchの対象と限界

### 対象となるもの

| 対象 | 説明 |
|------|------|
| **Linuxカーネルの脆弱性** | Critical / High のCVSSスコア |
| **特権昇格（Privilege Escalation）** | 一般ユーザーがroot権限を取得できる脆弱性 |
| **リモートコード実行（RCE）** | ネットワーク越しにコードを実行される脆弱性 |

### 対象外のもの

| 対象外 | 理由 |
|--------|------|
| **ユーザースペースのライブラリ** | OpenSSL、glibcなどは `unattended-upgrades` が担当 |
| **Low / Medium の脆弱性** | 通常の apt upgrade で対応 |
| **カーネル以外のすべて** | Livepatchはカーネル専用の仕組み |

:::caution[よくある誤解]
Livepatchは「すべてのセキュリティパッチを再起動なしで適用する」ものではありません。**カーネルのCritical/High脆弱性だけ** が対象です。それ以外のパッケージ更新には通常の `apt upgrade` が必要です。
:::

## Livepatchは「再起動不要」を意味するのか？

### 答え: NO

Livepatchは **再起動の代替ではなく、計画外再起動を防ぐツール** です。

```
再起動が必要な理由（Livepatchがあっても）:

1. メモリリークの蓄積
   └─ 長期稼働するとメモリ上にゴミが溜まる

2. ファイルハンドルのリーク
   └─ 使い終わったファイルの参照が残る

3. Medium/Low の脆弱性修正
   └─ Livepatchの対象外の修正を適用するため

4. カーネルのメジャーアップデート
   └─ 新機能や大幅な変更は再起動が必要
```

### 推奨される運用

```
Livepatch の正しい使い方:

定期メンテナンス日を設定（例: 月1回）
  │
  ├── 通常時: Livepatchが Critical/High を自動修正
  │           → 再起動なしで安全性を維持
  │
  └── メンテナンス日: 通常の apt upgrade + 再起動
                       → すべてのパッチを適用
                       → メモリ状態をクリーンに
```

## 家庭ユーザーにとっての実用価値

### メリット

1. **作業の中断を減らせる**  
   緊急のカーネル脆弱性が出ても、今やっている作業を中断して再起動する必要がない

2. **セキュリティの空白期間が短くなる**  
   「再起動面倒だからあとで…」という間にも脆弱性は修正済み

3. **設定不要で自動適用**  
   Livepatchを有効化すれば、あとは自動的にパッチが適用される

### 家庭ユーザーのよくある疑問

**Q: PCをシャットダウンしても大丈夫？**  
A: もちろん大丈夫です。Livepatchは「再起動を禁止する」ものではなく、「緊急の再起動を不要にする」ものです。

**Q: 再起動したらLivepatchのパッチは消える？**  
A: 再起動すると通常の apt upgrade で適用されたカーネルが起動します。Livepatchのパッチは再起動後のカーネルには含まれていますが、通常のパッケージとして適用された状態になります。

**Q: Livepatch自体の設定は必要？**  
A: `pro status` で enabled になっていれば、自動的に動作しています。特別な設定は不要です。

## まとめ

| 項目 | 内容 |
|------|------|
| **目的** | カーネルの重大な脆弱性を再起動なしで修正 |
| **対象** | Critical / High のカーネル脆弱性のみ |
| **方式** | 累積パッチ（安全性重視） |
| **自動化** | 有効化すれば完全自動 |
| **注意点** | 再起動の代替ではない。定期的な再起動は必要 |
| **家庭ユーザー** | 作業中断の削減、セキュリティ空白期間の短縮 |
